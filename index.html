EARLY CAREER ARTIFACTS

Disaster Recovery (DR) Activity for a Business-Critical SAP Application

Introduction

As part of my role in technical support and maintenance of a business-critical enterprise application, I was entrusted with planning and executing the DR activity — validating the organization's ability to switch application and database operations from the Primary Data Center to a geographically separate DR site in the event of a disaster.

Description

The application carried a strict Recovery Time Objective (RTO) requiring restoration within a few hours of a declared disaster. My responsibility included the technical execution of the failover — coordinating cutover of application and database services from Primary to DR servers — first validated on Development, then executed on the larger, more complex Production environment.

Process & Challenges
  • Coordinated with SAP PO Support, Cloud Operations, Application Development, and DNS teams to define roles, timelines, and dependencies.
  • Executed the DR failover on Development as a successful proof of concept.
  • SAP Channel Conflict: separate Primary/DR channels caused message failures — resolved by reconfiguring to a single unified channel that routed correctly to whichever server was active.
  • Disk Tier Limitations: Production's larger data volume exposed an I/O bottleneck — resolved by upgrading to a premium storage tier.
  • Carefully reconfigured application-to-database connectivity post-cutover, then verified availability and data continuity, obtaining client sign-off.
Tools & Technologies

SAP Process Orchestration, SAP Communication Channels, Application/Database Server Administration, Cloud Storage Management, DNS Configuration.

Value Proposition

Unique Value: Independently troubleshooting production-scale issues that didn't surface in lower environments, including designing a lasting fix for the SAP channel conflict with subject matter experts, while owning a high-pressure, cross-team activity.

Relevance: Directly applicable to infrastructure resilience, cross-team technical coordination, SAP administration, and production support under strict RTO commitments.

References

SSL Certificate Migration: Self-Signed to Commercial CA

Introduction

The business-critical application I supported was originally secured with a self-signed SSL certificate, introducing trust and security concerns for an internet/browser-facing application. I was assigned to lead migration to a certificate from a trusted commercial CA — a task new to me at the time.

Description

Self-signed certificates lack external CA validation, creating browser trust warnings, exposure to man-in-the-middle/spoofing risks, and operational overhead. I independently procured the certificate through DigiCert and led coordination for deploying it across the Production landscape, including the Web Application Firewall (WAF) and backend Application servers.

Process & Challenges
  • Procured the certificate via the DigiCert portal, issued within a few days.
  • Validated the certificate on Development, running an upgraded application version, without issues.
  • Risk: Production ran an outdated, unpatched version prone to instability during changes.
  • Mitigation: pre-imported the certificate chain and scheduled the cutover for a low-traffic weekend maintenance window.
  • Coordinated activation with the Server Maintenance team updating the WAF and Application servers in parallel.
  • Post-update outage: the application became inaccessible via browser after the cert update; the Application Development team's backend restart didn't resolve it — I performed a restart of the internal application services myself, which fixed the issue.
Tools & Technologies

DigiCert (Commercial CA) portal, SSL/TLS Certificate Management, Web Application Firewall, Application Server Administration, cross-functional coordination with Server Maintenance and Application Development teams.

Value Proposition

Unique Value: Taking ownership of an unfamiliar technical task end-to-end with minimal prior experience, paired with sound risk management — planning around an unstable Production version and staying engaged through live troubleshooting until fully resolved.

Relevance: A common, critical security remediation directly relevant to application security, SSL/TLS lifecycle management, production change management, and time-boxed incident troubleshooting.

References

Internal Change Request / Maintenance Window Documentation for the Certificate Migration.
